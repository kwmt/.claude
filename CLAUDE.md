## Git Workflow Rules

plan modeから実行モードに切り替える際、現在のブランチが main, develop, staging のいずれかである場合:
1. 作業開始前に新しいブランチを作成する（例: feat/xxx, fix/xxx）
2. 作業完了後にPRを作成する（/create-branch-pr または /pr コマンドを使用）

## Agent Team Rules

実装タスクを行う際は、以下の役割でエージェントチームを編成すること。
PMがタスクの規模・複雑さに応じて必要な役割を柔軟に選択する（全役割は推奨構成）。

### チーム構成

| 役割 | 説明 |
|------|------|
| **Project Manager (Tech Lead)** | チームリーダー。タスク分割・依存関係管理・進捗管理。チーム規模の判断も担当 |
| **実装者** | 実際のコード実装を担当。タスクの規模に応じて複数人配置可 |
| **テスト担当** | ユニットテスト・統合テストの作成を担当 |
| **QAエンジニア** | 実装完了後の品質検証・動作確認を担当 |
| **レビュワー** | コードレビュー（セキュリティ・パフォーマンス・保守性）を担当 |
| **悪魔の代弁者** | 設計フェーズと実装後レビューの両方で、批判的視点から問題点・リスク・改善案を提示 |

### 悪魔の代弁者の介入タイミング
1. **設計フェーズ**: 実装方針・アーキテクチャ決定時に批判的レビューを実施
2. **実装後レビュー**: レビュワーと共にコード・設計の問題点を指摘
